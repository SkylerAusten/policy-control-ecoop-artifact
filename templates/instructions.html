Access Control Policies Study

Access Control Policies are designed to allow or block requests for resources. In this study, you'll be shown a description of an access control policy in English. This description was given to an AI language model to generate candidate policies that might match that description.

Request Components

Each request has a:

  • Subject (sub): the person making a request. Subjects have one or more roles (e.g., Lawyer, Manager).
  • Action (act): what the subject wants to do (e.g., Read, Write).
  • Resource (res): the thing being accessed (e.g., MedicalRecord, FinancialReport).

Subjects, Actions, and Resources may have attributes that further specify their characteristics (e.g. sub.inTraining = True).

Example Policy

Let's now look at an example policy...

Policy Description:

Managers can...

- read any resource.

- write payslips, unless they're in training.

Lawyers can...

- read and write contracts.


Corresponding Policy:

permits = { 
  (role = Manager and act = Read) or
  (role = Manager and act = Write and res = Payslip) or
  (role = Lawyer and res = Contract)
}

denies  = {
  (role = Manager and act = Write and res = Payslip and sub.inTraining = True)
}

Anything not expressed in the permit rules is implicitly denied.

How to Read a Request

Subject: Person1
  roles: {Lawyer}
  inTraining: False
Action: Read
Resource: Contract

This represents a lawyer (not in training) requesting to read a contract.

How to Evaluate Requests

When evaluating a request:

  1. The subject is mapped to their role(s).
  2. For each role, the request is evaluated against the permit and deny rules:
    • If no permit rule applies, the request is denied (by default) for that role.
    • If a permit rule applies but a deny rule also applies, the request is denied for that role.
    • If a permit rule applies and no deny rule applies, only then is the request permitted for that role.
  3. A request is allowed if it is permitted by at least one of its roles. Otherwise, it's blocked.

Remember to evaluate the policy and requests against the policy description as written, not how you think the policy should work.

These requests should be ACCEPTED
Subject: Person1
  roles: {Lawyer}
  inTraining: False
Action: Read
Resource: Contract

This request is allowed. Lawyers are permitted to read contracts, and there is no deny rule that applies. 

Subject: Person3
  roles: {Lawyer, Manager}
  inTraining: True
Action: Read
Resource: Payslip

This request is allowed. Managers are permitted to read any resource, and the deny rule only restricts writing payslips (while in training)—not reading them. The fact that lawyers are implicitly denied when reading payslips doesn't matter, because at least one role (manager) is permitted.

These requests should be REJECTED 
Subject: Person2
  roles: {Manager}
  inTraining: True
Action: Write
Resource: Payslip

This request is blocked. While managers are generally permitted to write payslips, the deny rule explicitly prevents managers who are in training from writing payslips.

Subject: Person3
  roles: {Lawyer}
  inTraining: False
Action: Write
Resource: Payslip

This request is blocked. Lawyers are only permitted to access contracts. Since there is no permit rule for lawyers writing payslips, the request is denied by default.
